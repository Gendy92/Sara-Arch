This is a classic check-then-act race. If your `profiles` table has a database trigger that fires on `auth.users` insert (a very common Supabase setup â€” something like `handle_new_user()` that auto-creates a profile row), then the moment `admin_create_auth_user` succeeds, a profile may already exist by the time this JS resumes execution and runs its `GET`. If there's any replication delay between the write (trigger insert) and the read (PostgREST GET), the `GET` can return `existing.length === 0` even though a row is about to exist or already does, and your code chooses POST, colliding with the trigger's insert. This is the single most likely explanation for the 409 you saw, and it's not something you can fix by retrying or waiting â€” you need to either remove the trigger, or stop doing a separate INSERT and use upsert semantics instead (`POST` with a `Prefer: resolution=merge-duplicates` header, i.e. an actual database-level upsert) so there's no race window at all.
